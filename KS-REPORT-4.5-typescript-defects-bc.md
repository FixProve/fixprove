## 3. Defects caught and fixed (specific, not summarized)

### Defect B — Resolver cannot trace a `new NamedImportClass(...)` instance back to its import (fixed, revised scope)

**Symptom (carried forward from Session 4.4):** `const resend = new Resend(apiKey); resend.emails.sendBulkWithRetry(...)` — a hallucinated instance method reached through a `new`-constructed local variable — was never flagged, because `ts_resolver.py`'s Pass B only checked `default_import`/`namespace_import` bindings, and local variables were never tracked at all.

**Design decision (Yehor's explicit sign-off):** direct assignment only. `ts_symbol_extractor.py` now recognizes `const/let/var x = new Foo(...)` where the declarator's name is a plain identifier (not a destructuring pattern) and the constructor is a plain identifier (not `new ns.Foo()`) — captured as a new `new_bindings` output category. A later plain reassignment (`let x; x = new Foo()`) or aliasing through another variable (`const y = x`) is deliberately NOT tracked; both fall out naturally from the AST shape (a `variable_declarator` vs. an `assignment_expression`) without extra code, rather than requiring active exclusion logic.

**Second design decision (Yehor's explicit sign-off, surfaced mid-design):** resolution depth stays at ONE HOP, uniform with every other binding kind, even though this means the *original* Defect B seeded case is a two-hop chain (`resend` → `.emails` [real property, hop 1] → `.sendBulkWithRetry` [hallucinated, hop 2]) that this fix structurally cannot catch. Catching it would require also flattening the *type* of every hop-1 property — a materially bigger change, and one that would make instance bindings inconsistent with every other binding kind's depth rule. What the fix DOES catch: a hallucinated method or property directly on the instance (`resend.notAMethod()`), exactly mirroring how `axios.getData()` (hop-1 hallucinated method on a default-import instance) is already caught.

**Implementation:** `ts_knowledge_base.py`'s `build_package_entry` now produces `class_member_symbols`, a `{class_name: [flattened_members]}` map for every exported class, reusing the same extends-chain flattening already proven against axios's 3-level chain — just keyed per class instead of only for the default-export target. `ts_resolver.py`'s `_build_alias_map` cross-references `new_bindings` against named-import local names to produce a `named_import_instance` binding kind, checked at Pass B with the same one-hop logic as `default_import`/`namespace_import`.

**Verified by:** 9 new symbol-extractor tests (including a property-based test asserting the direct-assignment-only contract holds across 200 randomly generated declaration shapes), 4 new KB tests, 6 new resolver tests including one that explicitly reproduces the *original* two-hop seeded case and confirms it is — correctly, by design — NOT flagged, distinguishing "not yet implemented" from "structurally out of scope, decided."

### Defect C — `@types/react`'s `export = Namespace` shape false-positives every hook/type import (fixed, narrow scope)

**Symptom (carried forward from Session 4.4):** `import { useState, useEffect, Component, ReactNode, FormEvent } from "react"` — all real, valid named imports — were flagged as unresolved, because `@types/react`'s `export = React;` targets a `declare namespace React {...}`, and the KB's `export =` handling only ever populated `default_export_symbols` (for default-import member access), never `exported_names` (for named-import checks).

**Design decision (Yehor's explicit sign-off):** namespace-confirmed only, not universal `export =` flattening. `ts_knowledge_base.py` now records `declare namespace X {...}` blocks into a new `namespaces` dict (name → set of un-prefixed member names, existence-only, not member-flattened) via a new `_namespace_member_names` helper. `build_package_entry` flattens a namespace's members into `exported_names` ONLY when `export =`'s target resolves to a confirmed entry in that `namespaces` dict — `export = SomeClass` / `export = SomeInterface` are completely unaffected. This is deliberately narrower than flattening every `export =` target, consistent with the KB's standing "ambiguity resolves to silence" principle: an unconfirmed shape stays silent rather than being guessed at.

**Implementation:** the `internal_module` tree-sitter node type (namespace) is now recognized as a distinct case inside `_parse_declaration`'s `ambient_declaration` branch, separate from `module` (module augmentation) — confirmed via a direct grammar probe that these are genuinely different node types, so a namespace can never be mistaken for the module-augmentation degradation trigger.

**Verified by:** 4 new KB tests (namespace member parsing, the namespace/module-augmentation non-confusion guard, the react-shaped end-to-end case, and the class-not-namespace regression guard that caught Defect D — see below) plus 2 new resolver tests reproducing the exact live-verified regression from `yehor.ai#1`'s second seeded-PR run.

### Defect D — `export = Foo` where Foo is a class declared directly (not via a const alias) loses `default_export_symbols` (found and fixed this session, pre-existing gap)

**Symptom:** discovered via this session's OWN regression test for Defect C's scope boundary (`test_export_equals_class_not_namespace_unaffected`): `declare class Foo { bar(): void; } export = Foo;` produced `default_export_symbols: None` instead of `["bar"]` — a real, valid shape silently losing verifiability.

**Root cause:** `build_package_entry`'s `type_name` resolution for `export =`/`export default` only ever checked `merged.default_ref in merged.interfaces` as a fallback (for the case where the default-ref names a type directly rather than through a `declare const x: Foo` alias) — it never checked `merged.classes`. This is a pre-existing gap in Session 1.4's original code (the axios/fp-ts-clean-demo fixtures that established this logic both export a *const instance* of a type, not a class name directly), newly exposed by this session's own adversarial test — not a Defect B/C regression, and not something either defect's design decisions touched.

**Fix:** added `if type_name is None and merged.default_ref in merged.classes: type_name = merged.default_ref`, mirroring the existing interfaces-only check, one line, same pattern already established.

**Verified by:** the same regression test that found it now passes; the fix required no design sign-off given its narrow, single-line, same-pattern nature — flagged to Yehor in this report as a found-and-fixed defect per the standing "document every defect found and the exact fix applied" rule, not pre-approved as an architectural decision.

2. **Defect B's original seeded case remains uncaught, by design.** `resend.emails.sendBulkWithRetry(...)` (two hops past the binding) is NOT caught by this session's fix — only a hop-1 hallucination directly on an instance is. This is a deliberate, Yehor-approved scope boundary, not an oversight, but it means any TypeScript codebase with a deeply-nested hallucinated call through an instance property remains a false negative.
3. **Namespace member collection is name-only, not recursively flattened.** A class declared *inside* a `declare namespace {...}` block (e.g. react's own `Component`) is now a valid named-import target (Defect C fixed), but instantiating it via `new` and calling a hallucinated instance method on it would NOT be caught (`class_member_symbols` has no entry for namespace-nested classes) — an accepted, narrower scope boundary per the module docstring, not yet exercised by a real seeded case.

## 6. Methodology note — one suggested improvement to the process itself

This session's most valuable adversarial-verification moment came from a test that was never meant to find a defect: `test_export_equals_class_not_namespace_unaffected` was written purely as a regression guard for Defect C's scope boundary (making sure `export = SomeClass` didn't get accidentally broadened), and it incidentally exposed Defect D, a genuine pre-existing gap unrelated to either defect this session was scoped to fix. Worth deliberately generalizing as a practice: when writing a "must NOT change" regression test for a narrow fix, deliberately pick a realistic-but-previously-unexercised shape for the negative case (here: a class-not-namespace `export =`, a shape no existing test happened to cover) rather than the simplest possible non-example — the narrow regression test is free real estate for catching adjacent, unrelated bugs the same way the "flagship" test catches the one it was designed for.
