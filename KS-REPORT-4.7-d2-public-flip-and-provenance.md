Fixed: `git pull` now fast-forwards cleanly with no warnings, confirmed twice (once after the D2-flip PR merge, once after the metadata-fix PR merge).
- **`v0.1.7` release — real defect caught, not glossed over:** the actual `publish-npm` step log was retrieved and read in full (not inferred from the job's red/green status alone). Confirmed via direct fetch of `pypi.org/project/fixprove` that PyPI's own "successful" job that run was a silent no-op (still showing `0.1.6`, unchanged, dated Jul 9) — the version had never been bumped, so nothing was actually published under `v0.1.7` on either registry. - **`v0.1.8` release, attempt 1 — second real defect caught:** `publish-pypi` succeeded and was independently verified live (`pypi.org` showing `fixprove 0.1.8`, released `2026-07-13 16:23:07` — a genuinely new entry, not a repeat). `publish-npm` failed with a different, new error (`E404`) than the one just fixed; root-caused against `npmjs.com/settings/fixprove/tokens` directly rather than guessed — the token backing `NPM_TOKEN` (`fixprove-release-ci v2`) was confirmed showing status "Expired," with an expiry date of the same day as the failure.
- **`v0.1.8` release, attempt 2 — full independent verification:** after the token was rotated and only the failed `publish-npm` job was re-run (confirmed via the re-run dialog that `publish-pypi` was correctly excluded, avoiding a duplicate-publish failure), the job log was read in full and shows `+ fixprove@0.1.8` and a genuine Sigstore transparency-log entry (`logIndex=2163744385`, distinct from the earlier failed attempt's `2163710126`). - **Tools used:** direct git history scan (`git log`/`grep` across all refs), `git for-each-ref`/`packed-refs` inspection, live GitHub Actions log retrieval via Claude-in-Chrome (`get_page_text` on job pages, since GitHub gates full logs behind sign-in even for public repos and generic `web_fetch` cannot authenticate), direct live-registry checks against both `pypi.org` and `npmjs.com` (not just their respective CI job statuses), YAML/JSON parse validation (`python3 -c "import yaml/json"`) on every edited file, and the established write-to-new-name-then-verify-via-fresh-Read protocol for every file edit on the FixProve mount.

## 3. Defects caught and fixed (specific, not summarized)

### Defect E — npm provenance rejects on case-sensitive repository URL mismatch (`v0.1.7` failure)

**Symptom:** `npm publish --access public --provenance` failed with `E422`: `Error verifying sigstore provenance bundle: Failed to validate repository information: package.json: "repository.url" is "git+https://github.com/fixprove/fixprove.git", expected to match "https://github.com/FixProve/fixprove" from provenance.`

**Root cause:** `cli/package.json`'s `repository.url` used lowercase `fixprove/fixprove`; the actual GitHub org/repo is `FixProve/fixprove` (capital F, capital P). npm's Sigstore provenance verification is case-sensitive against the real OIDC claim from the Actions run. This defect existed in the file for the entire project history but was invisible until this session, because this was the first time `--provenance` had ever reached the verification step at all — every prior attempt (Session 4.2 onward) failed earlier, on the private-repo E422, before this code path was ever exercised.

**Fix:** corrected the casing in `cli/package.json`. Also found and fixed the identical casing bug in `engine/python/pyproject.toml`'s `Repository` field — this one had not caused a hard failure (PyPI's provenance check does not cross-validate repository casing the same way), but was the same wrong metadata, fixed for correctness while the file was already being touched for the version bump.

**Test:** live `v0.1.8` release run — `publish-npm` job's own log, `npm notice publish Signed provenance statement... Provenance statement published to transparency log`, followed by a subsequent, distinct failure (Defect F, below) rather than a repeat of this one — confirming the case-mismatch class of failure is genuinely resolved, not just suppressed.

### Defect F — expired npm access token surfaces as a misleading `E404`, not `E401`/`E403` (`v0.1.8` attempt 1 failure)

**Symptom:** after Defect E's fix, `npm publish --access public --provenance` progressed further — successfully signed the provenance statement and published it to Sigstore's transparency log — then failed at the final registry upload with `E404`: `404 Not Found - PUT https://registry.npmjs.org/fixprove - Not found ... 'fixprove@0.1.8' is not in this registry.` This is a misleading error shape for a package that had published five times previously without incident; a naive read would suggest a missing/misnamed package, not an auth problem.

**Root cause:** the access token backing the `NPM_TOKEN` repository secret (`fixprove-release-ci v2`, a bypass-2FA Automation token created Session 4.2/later-rotated) had a 7-day expiration window and had expired that same day, confirmed directly on `npmjs.com/settings/fixprove/tokens` (status: "Expired," expiry date matching the failure date exactly). npm's registry returns 404 rather than a clean 401/403 for certain auth-permission failures on provenance-enabled publishes — likely deliberate, to avoid confirming package existence/ownership to an unauthorized caller — which made this look like a different class of bug than it was.

**Fix:** Yehor generated a new Automation (bypass-2FA) token scoped specifically to the `fixprove` package with Read/write permission and a longer (90-day) expiration, updated the `NPM_TOKEN` GitHub secret directly, then re-ran only the failed `publish-npm` job (not the whole workflow, to avoid `publish-pypi` attempting — and failing — a duplicate publish of the already-live `0.1.8`).

**Test:** re-run job log shows `+ fixprove@0.1.8` and a new, distinct Sigstore transparency-log entry; independently cross-checked live on `npmjs.com/package/fixprove` (version `0.1.8`, published minutes prior, provenance badge present) rather than trusting the job's green status alone.

### Item #7 (carried forward, now genuinely fixed, not just diagnosed) — two 0-byte broken tag refs blocking `git pull`

**Symptom:** `git pull` on Yehor's own machine failed with `fatal: bad object refs/tags/v0.1.4.lock.bak.1783353772` / `error: ... did not send all necessary objects` — confirming this pre-existing known issue (documented since Session 4.5) had escalated from a cosmetic sandbox curiosity to an active blocker of normal git operations on the real, live repository.

**Root cause:** `refs/tags/v0.1.4.lock.bak.1783353772` and `refs/tags/v0.1.4.lock.bak.1783353789` were confirmed to be exactly 0 bytes on disk — invalid as ref files by any definition (a ref must contain a 40-char SHA or symref pointer). `packed-refs` was checked directly and holds no entry for either name, confirming these were pure loose-file artifacts with no packed fallback to protect, most likely a stale/duplicated lock-file remnant from a cloud-sync tool race, matching the `.git/index.lock` corruption class already documented for this mount.

**Fix:** the two empty files were deleted directly (`Remove-Item ... -Force`) after confirming via `packed-refs` that nothing legitimate could be lost. `git pull` now fast-forwards cleanly, confirmed on two separate merges this session.

## 4. Known limitations (stated honestly)

1. **npm's own account UI is actively warning that bypass-2FA tokens are being restricted** — partial restriction from Aug 2026, full elimination of direct-publish-via-bypass-2FA-token by Jan 2027. This pipeline's npm publishing was deliberately built on exactly this token type (Session 4.2, to enable non-interactive CI). Yehor explicitly chose to rotate the token and defer a migration to npm Trusted Publishing (OIDC, mirroring what's already live for PyPI) to a dedicated future session, rather than scope-creep this one. **This is a real, time-boxed architectural item, not indefinitely deferrable** — it should be picked up well before Jan 2027.
2. **The new `NPM_TOKEN`'s expiration (90 days from today, per Yehor's stated choice during token creation) is not tracked anywhere durable** — it lives only in GitHub's secret store, which does not surface expiry warnings the way `npmjs.com`'s own token list does. Recommend either a calendar reminder or picking a longer-lived token next rotation, given the above Trusted Publishing migration is the real fix.
3. **`v0.1.7` is a permanently burned, dead tag** — it exists in the repository's history but nothing was ever published under it on either registry. This is intentional (Yehor's explicit tag-immutability decision, Section 1), documented here so a future session doesn't mistake its existence for a real, if broken, release.
4. **`release.yml`'s public-repo precondition for `--provenance` is enforced by process/documentation only**, not by a CI-side check. If the repository were ever made private again (no current plan to do so) and a tag pushed, `publish-npm` would fail with the same `E422` as `v0.1.7`, and nothing in the workflow itself would pre-empt or explain that failure beyond the KS-TRACE comment.
5. **All items carried forward from Session 4.6's own Known Limitations that this session did not touch remain open**: Defect B's two-hop instance-tracking architectural question, Worker `push`-event Check Run correlation, `build_knowledge_base`'s cache-invalidation gap, `npx wrangler` on 3.114.17, the untracked `logo/` directory, and `autonomous-core`'s untracked-file pile — none were in this session's scope and none were touched.
6. **Mid-session, Claude-in-Chrome was transiently unreachable** and recovered without any user-side fix being identified — noted in case it recurs; it did not block the session (GitHub's public-repo pages were readable via direct fetch in the interim, though full Actions logs required the browser tool once it reconnected, since GitHub gates those behind sign-in even for public repos).

## 6. Methodology note — one suggested improvement to the process itself

Twice this session, a green CI job status would have been the wrong signal to trust in isolation: `v0.1.7`'s `publish-pypi` job passed but was a silent no-op (version never bumped, nothing new published); `v0.1.8`'s first `publish-npm` attempt failed with an error shape (`E404`) that, read in isolation, suggests a missing-package problem rather than the actual expired-token cause. In both cases, checking the live registry (`pypi.org`, `npmjs.com`) directly — independent of what GitHub Actions reported — was what actually surfaced the truth. Worth generalizing beyond this project: for any pipeline whose job is to publish to an external system, "the job went green" and "the job's error text" are both secondary evidence; the primary source of truth is always the external system's own live state, and it's worth the extra step of checking it directly rather than trusting either the CI status or the CI's own error message at face value.
