**Architectural decisions made this session (logged as they happened, each with Yehor's explicit sign-off before the triggering commit/tag/push):**

1. **Static fallback table vs. pinning PyYAML as a test dependency** (v0.1.1). Rejected the narrower fix (just installing PyYAML so the test passes) because it would fix the test without fixing the real gap: a customer with an uninstalled, name-mismatched dependency would never get a "dependency-not-installed" finding. The static table only activates when the dynamic, real-environment resolution has nothing to say, and never overrides real installed-package data (verified by a dedicated test).
2. **New local fixture package (`fp-monkeypatch-demo`) rather than a real hard-exiting third-party package** (v0.1.1). No real PyPI package reliably hard-exits at import time on demand; built as a proper local `pyproject.toml` package mirroring the already-established `ts_corpus/local_fixtures/` pattern.
3. **Drop `npm --provenance` rather than delay the npm release or flip the repo public early** (v0.1.3, Yehor's explicit decision). Rationale, in Yehor's own words: live end-to-end testing of the GitHub Action template on private test repos could not wait, and prematurely flipping the repo public would short-circuit the deliberate, separate D2 launch decision. Logged as a temporary, explicitly-approved trade-off with a standing requirement to restore `--provenance` immediately before the D2 public flip — this is carried forward as an open item (Section 4).
4. **PyPI Trusted Publisher entry: delete-and-recreate rather than debug the existing entry** (external action, between v0.1.1 and v0.1.2). The existing entry's fields *appeared* correct in a clear screenshot (`FixProve/fixprove`, `release.yml`, `pypi`) but PyPI still rejected the OIDC exchange as an invalid-publisher mismatch. Root cause was never conclusively identified (most likely invisible whitespace from a prior paste); resolved pragmatically by having Yehor delete the entry and retype it fresh rather than continuing to debug a config that looked correct. This is logged honestly as an unresolved root cause, not a false claim of full diagnosis.

PyYAML is correctly not a runtime dependency, so this is a genuine false-negative risk for any uninstalled, name-mismatched dependency. | Static fallback table `_KNOWN_DIST_TO_IMPORT_NAME_MISMATCHES`, consulted only when the dynamic map has nothing; dynamic data always wins when present. | 185/185 tests pass; dedicated "dynamic wins" test |

## 4. Known Limitations (stated honestly, nothing softened)

1. **Defect #4 (PyPI Trusted Publisher invalid-publisher) was never conclusively root-caused.** The fix (delete and recreate the entry) worked, but *why* the original, visually-correct entry was rejected remains unknown. If this recurs, don't assume the same "retype it" fix will work a second time — it may mask a different underlying cause.
2. **`npm --provenance` is currently absent from the release workflow**, per Yehor's explicit, temporary decision. This must be restored before the D2 public-repo flip (Section 1, item 3) — flagged again here so it is not silently forgotten. No mechanism currently exists in the repo to *enforce* this restoration happens; it depends on a human or a future AI session remembering to check this report.
3. **The npm side of `v0.1.4`'s pipeline run "failed"** in GitHub's UI (red X on the overall run), which could be misread as an outstanding defect. It is not — see Section 2's explanation. A future session picking up this repo should not attempt to "fix" this without first checking whether npm's currently-published version is what's actually being republished.
4. **No real npm publish has been attempted since the `--provenance` removal with an actual version bump.** The `--provenance`-free path has only been proven to work for the exact `0.1.0` version that was already live; a future release (e.g. `0.1.1` with real code changes) will be the first genuine test of the current publish-npm step end-to-end.
5. **`ci.yml` (regular push/PR CI) still does not run the Python test suite at all** — only `release.yml` (tag-triggered) does. Pre-existing gap from Session 4.1, not fixed this session (out of scope), flagged again as a suggested follow-up (Section 6).
6. **The static distribution→import-name mismatch table (Defect #1's fix) is not exhaustive** — 14 well-known cases; an unlisted uninstalled/mismatched dependency still falls back to the pre-existing best-effort guess.
7. **This sandbox's mount exhibited real, only-partially-understood git lock/rename reliability problems this session** (Defect #10), serious enough to require independent verification after every single git write operation. This is a constraint of the current AI working environment, not of the FixProve repository itself, but is worth any future session's awareness if working from this same sandbox.
8. **`logo/` appears as an untracked directory** in the working tree as of this report — not created or touched by this session's work; origin not investigated (out of scope for this session). Flagged for Yehor's awareness, not acted on.

## 6. Methodology Note — One Suggested Improvement

This session is the clearest evidence yet, in this project's own history, for why Keystone Stage 3 insists on live verification rather than local/simulated checks: of the ten defects logged in Section 3, eight (#3 through #10) were entirely invisible to any local build, test, or lint step and were discovered only by an actual push to real infrastructure — often only after a *previous* fix let the pipeline progress one step further into territory no prior session had ever reached. `twine check`, `npm pack --dry-run`, and a full green local `pytest` run all say nothing about GitHub's `permissions:` inheritance semantics, PyPI's Trusted Publisher OIDC matching, PyPI's comma-delimited Project-URL parsing, PyPI's 32-character label limit, or npm's provenance/repo-visibility coupling. Suggested improvement: for any project doing its *first* real release to a package registry (as opposed to the Nth), the Keystone contract for that specific session should explicitly budget for "the live pipeline will very likely fail multiple times on infrastructure-specific constraints no local check can see" as an expected, planned cost — not a sign the local verification work was inadequate — so that neither the AI nor the Director treats each new live-only defect as a surprise requiring re-litigating the plan, and so that time/session-count estimates for "first live release" work account for this pattern rather than assuming one clean pass.
