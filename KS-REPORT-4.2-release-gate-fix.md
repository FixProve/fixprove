**Files delivered/modified this session:**

- **`engine/python/knowledge_base.py` (modified)** — added `_KNOWN_DIST_TO_IMPORT_NAME_MISMATCHES`, a static table of well-known distribution→import-name mismatches (`pyyaml`→`yaml`, `beautifulsoup4`→`bs4`, `pillow`→`PIL`, etc.), consulted in `_resolve_import_name` only as a fallback when the dynamic, installed-package-derived `reverse_map` has no candidate.
- **`engine/python/tests/test_knowledge_base.py` (modified)** — 3 new regression tests + 1 hypothesis property test for the changed logic.
- **`engine/python/corpus/local_fixtures/fp-monkeypatch-demo/` (new)** — a tiny, real, pip-installable local package (mirrors the existing `ts_corpus/local_fixtures/` pattern) that hard-exits at import time, used by the corpus eval's "degraded package" clean case.
- **`engine/python/corpus/NOTES.md` (new)** — documents the corpus's real fixture-dependency setup (mirrors `ts_corpus/NOTES.md`).
- **`.github/workflows/release.yml` (modified)** — the `test` job now explicitly installs the corpus eval's real fixture dependencies (`pandas`, `requests`, the local `fp-monkeypatch-demo` fixture, and `ts_corpus`'s npm packages via `npm ci`) before running `pytest`.

**Architectural decisions made without stopping to ask (logged here, pending Yehor's explicit sign-off before this is committed/tagged):**

1. **Static fallback table vs. just pinning PyYAML as a test dependency.** The failing test could have been made to pass by simply adding `PyYAML` to `engine/python/requirements.txt` so it's actually installed wherever tests run. I rejected that as insufficient: it would only fix the *test*, not the underlying real-world gap it exposed — a customer whose `requirements.txt` pins `PyYAML` but who has **not actually installed it** (exactly the "dependency-not-installed" case this tool exists to catch) would have that dependency silently un-checked entirely, because the import-name bridge could never learn that `yaml` maps to `PyYAML` without the package being installed to introspect. I chose the static-table fallback specifically because it fixes the deeper case, not just the test, and only activates when the dynamic (real, live-environment) resolution has nothing to say — it can never override real installed-package data.
2. **New local fixture package (`fp-monkeypatch-demo`) rather than pinning a real, hard-exiting third-party package.** No real PyPI package reliably hard-exits at import time on demand, so Session 1.3's corpus eval always depended on an ad hoc local install that was never captured as a reproducible artifact anywhere in the repo — a genuine, previously-invisible gap. I built it as a proper local `pyproject.toml` package (mirroring the already-established `ts_corpus/local_fixtures/` pattern for the TS/JS side) rather than any lighter-weight approach, for consistency with existing, already-approved project conventions.
3. **CI fix lands in `release.yml` only, not `ci.yml`.** `ci.yml` (regular push/PR CI) does not run the Python test suite at all currently (confirmed in Session 4.1's log) — this was already a known, separately-flagged gap, not something introduced or fixed this session. I did not expand scope to fix it, per the explicit instruction to stay focused on the release pipeline. Flagged again in Section 6 as a suggested follow-up.

## 3. Defects Caught and Fixed (specific, not summarized)

## 4. Known Limitations (stated honestly, nothing softened)

1. **This session's local verification runs were done in throwaway `/tmp` copies and fresh venvs, not the actual GitHub Actions runner.** They are the closest reproduction achievable from this environment (fresh venv, fresh `npm ci`, no pre-existing cache — matching a real clean-checkout CI run step-for-step against the exact commands `release.yml` now runs) but are not a substitute for actually watching the real `v0.1.1` tag's pipeline run green on GitHub's own infrastructure. That is the next, unavoidable step.
2. **`ci.yml` (regular push/PR CI) still does not run the Python test suite at all** — only `release.yml` (tag-triggered) does. This means a regular PR could silently break the Python engine and nobody would know until the next tag push. This is a pre-existing gap (documented in Session 4.1's log), not introduced or fixed this session — flagged again as a suggested improvement (Section 6), not fixed here, since it's out of the immediate scope Yehor set for this session.
3. **The static distribution→import-name mismatch table is not exhaustive.** It covers 14 well-known cases. An uninstalled, name-mismatched dependency NOT in this table still falls back to the pre-existing best-effort normalized guess (same behavior as before this fix) — a real but unchanged limitation, not a regression.
4. **This session's sandbox mount exhibited real file-corruption and stale-lock behavior** (Section 3, Defect 4) serious enough that every edit in this session required a byte-for-byte `diff` re-verification step before being trusted. All delivered files have been verified this way, but this is worth Yehor's awareness if working in this same environment going forward — the failure mode is silent (tools report success) and would not be caught without an explicit re-read.
5. **Stray `.git/index.lock.bak*` files** were left behind in `.git/` this session as a side effect of working around the lock issue (Section 3, Defect 4). They do not affect git's operation (git only looks for the exact name `index.lock`) but are harmless clutter — safe to delete manually, not committed (`.git/` itself is never part of the working tree).
6. **Nothing has been committed, tagged, or pushed yet.** All changes described in this report exist only in the working directory pending Yehor's review of this report and explicit approval to proceed to Stage 5 (commit → tag `v0.1.1` → push → watch the real pipeline).

## 6. Methodology Note — One Suggested Improvement

Both root causes this session share a common shape: something that was true and verified in an earlier session's *local, ad hoc* environment was reported as "passing" without ever being captured as a reproducible artifact (a requirements file the CI actually installs from, a buildable fixture package, an `npm ci` step) that a later, genuinely clean environment could reconstruct. Session 1.3's own Keystone Report even flagged this precise risk in its Known Limitations ("Cold-cache CI performance is unresolved") without going one step further to ask "would this whole eval even *run correctly* cold?" — it would not have, and didn't, eight sessions later. Suggested improvement: whenever a Keystone Report's Verification Summary depends on "a genuinely pip/npm-installed real or local package," the report should be required to also state *how* a future clean environment (a new contributor's laptop, a CI runner, a different AI session) reproduces that exact installed state — ideally by pointing at a committed script or fixture, not by assuming the reader will re-derive the same ad hoc steps. This session's fix is exactly that: turning an implicit, undocumented environment assumption into an explicit, committed, CI-executed step.
