**What's explicitly NOT done — this is Session 4.5's primary mandate:**
1. **Defect B — resolver cannot trace a `new NamedImportClass(...)` instance back to its import.** `ts_resolver.py`'s Pass B only checks `default_import`/`namespace_import` bindings; a local variable assigned via `new SomeNamedImport(...)` never enters `alias_map`, so hallucinated instance methods on it are invisible. This is why the ORIGINAL seeded hallucination (`resend.emails.sendBulkWithRetry(...)`) went undetected on the first live run this session, and why the seeded test file was revised to a different pattern to still get useful signal on Defect A.
2. **Defect C — `@types/react`'s `export = Namespace` shape false-positives every hook/type import.** `react`'s types are declared inside `declare namespace React { ... }` with `export = React;` at module level; TypeScript treats namespace members as valid named-import targets under this pattern, but the KB's `export =` handling only ever populates `default_export_symbols` (default-import member access), never `exported_names` (named-import checks). **This is arguably the highest-impact of all three defects found across Sessions 4.3-4.4** — `import { useState } from "react"` is close to universal in real TS/JS codebases, so this defect would false-positive on the overwhelming majority of real-world React/TypeScript projects as the tool stands today.

Both are real, confirmed, live-verified architectural gaps — not hypothetical, not edge cases. TS/JS should not be represented as general-availability-ready until both are addressed.
