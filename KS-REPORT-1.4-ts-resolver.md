## 1. Provenance

**Locked architectural decision (Yehor, Session 1.4 Stage 1):** the TS/JS resolver is built with **tree-sitter-typescript + a custom Python knowledge-base builder**, explicitly rejecting the alternative I recommended (the real TypeScript Compiler API in Node). Yehor's choice keeps the engine in one language and one codebase, at the accepted cost of re-implementing (a strict subset of) what `tsc` already does for free — declaration merging, module augmentation, and full type resolution are NOT re-implemented; see Section 4.

Files delivered (under `engine/python/`, alongside Sessions 1.1–1.3):

- `finding.py` — **new shared module**, extracted from Session 1.3's `resolver.py`. Both the Python and TS/JS resolvers now import the same `dependency_status_finding` / `unresolved_symbol_finding` / `sort_findings` helpers, so "shares the deterministic core + output schema with the Python path" is a real code-sharing fact, not a coincidence of matching field names. `resolver.py` was refactored to use it (behavior unchanged — all 93 pre-existing Python-side tests still pass).
- `ts_symbol_extractor.py` (~370 lines) — tree-sitter-typescript AST extraction: imports (default/named/namespace/type-only/re-export/dynamic), call_targets, attribute_chains. Mirrors `symbol_extractor.py`'s contract, 1-indexed lines, overlap rule, and iterative (non-recursive) traversal.
- `ts_knowledge_base.py` (~430 lines) — parses installed npm packages' `.d.ts` files: locates entry points (bundled `types`/`typings` field, or `@types/<pkg>` fallback), follows triple-slash `/// <reference path>` directives and relative `export * from` / `export {x} from` re-exports transitively, flattens interface/class `extends` chains, and detects TypeScript module-augmentation (`declare module "..." { }`) to safely degrade rather than guess.

- `cli.py` — extended (not replaced) to discover `.ts`/`.tsx`/`.js`/`.jsx` files alongside `.py` files in the same pass, routing each to its own resolver and merging/sorting findings through the shared schema. A project with only one ecosystem's files is never blocked by the other ecosystem's missing manifest.
- `eval_ts_corpus.py` + `ts_corpus/` — 8 labelled TS files (5 clean, 3 hallucinated) run against **real, installed npm packages** (`axios` 1.18.1, `lodash` 4.18.1, `@types/lodash` 4.17.24) plus one hand-built synthetic fixture package (`fp-ts-clean-demo`) for re-export/extends-chain cases the sandbox's real packages didn't happen to exercise cleanly. `ts_corpus/NOTES.md` documents the one-time `npm install` setup step.
- `tests/test_ts_symbol_extractor.py` (26 tests), `tests/test_ts_knowledge_base.py` (30 tests, including one property-based test), `tests/test_ts_resolver.py` (25 tests), `tests/test_ts_cli_corpus.py` (7 tests) — 88 new tests.

## 3. Defects Caught and Fixed

**DEFECT KB-001 — `extends`-chain flattening missed CLASS-declared members, would have false-positived on real axios usage.**
Real axios's `.d.ts` declares `AxiosStatic extends AxiosInstance extends Axios`, but `Axios` (the interface that actually declares `.get`, `.post`, `.delete`, etc.) is a **class**, not an interface. The original member-collection helper only walked `interface_body` nodes; class-declared members were silently absent from the flattened default-export symbol set. Caught by a dedicated regression test built directly against the real installed axios package (not a mock) before this ever reached the resolver layer — the failure mode this would have caused (`axios.get(...)` flagged as hallucinated on completely valid code) is exactly the zero-false-positive violation this engine exists to prevent. Fixed by collecting members from `method_signature`/`property_signature`/`public_field_definition` uniformly across both `interface_body` and `class_body`.

**DEFECT KB-002 — multi-hop re-export chains silently dropped symbols from the middle/end of the chain.**
Found during the dedicated adversarial Verify pass (not a named acceptance case): `index.d.ts: export * from "./a"` where `a.d.ts` itself does `export * from "./b"` resolved only one level deep, silently dropping everything declared in `b.d.ts`. A legitimately re-exported symbol two or more hops from the entry file would have been an **unresolved-symbol false positive**. Fixed by making re-export resolution recursive (each re-export target's own re-exports are resolved before merging up), with a shared visited-paths set to guard against circular re-export chains (verified to terminate correctly on a constructed `A -> B -> A` cycle).

**DEFECT CORPUS-003 — the "not-installed" corpus fixture broke `npm install` for the delivered corpus.**
Found during final delivery verification (running `npm install` against the actual delivered `ts_corpus/package.json`, not just in the dev sandbox): a deliberately-fake package name (`totally-fake-fixprove-corpus-pkg`) was listed directly in `package.json`'s real `dependencies`, so npm correctly 404'd trying to resolve it — the corpus would have failed the very setup step its own `NOTES.md` instructs. Fixed by removing the fake name from `package.json` (kept fully install-able: `axios`, `lodash`, `@types/lodash`, plus two synthetic fixture packages wired in via `file:` dependencies) and injecting the fake name only at the KB-population layer inside `eval_ts_corpus.py`, where it's needed purely to exercise the not-installed test case. The CLI's own package-list derivation (strictly from a real project's `package.json`) never sees this name, which is correct, real-world behavior — `tests/test_ts_cli_corpus.py::test_cli_ts_only_project_exit_code_1` was updated to assert 2 CLI-visible findings (not 3), with the 3rd (`not-installed`) case covered by the dedicated corpus eval instead.

**Incidental finding (not this session's own code) — Session 1.1's property test had an incomplete keyword blocklist.**
While running the FULL regression suite (Python + TS together) as part of this session's delivery verification, `tests/test_symbol_extractor.py`'s `test_property_call_target_never_leaks_into_attribute_chain` (a Hypothesis property test from Session 1.1) failed on a shrunk example `chain="in"` — the test's hand-written keyword blocklist omitted `in` (and several other real Python keywords), so Hypothesis generated invalid Python (`in()`) and asserted an invariant about it that was never true. This is a latent defect in Session 1.1's test suite, not in `symbol_extractor.py`'s production code, and not part of this session's contract — but per Keystone's "don't just report the happy path" standard, it's disclosed here since it was found during this session's own work and directly affects a still-unsigned prior deliverable. Fixed by replacing the hand-maintained blocklist with Python's own `keyword.iskeyword`/`keyword.issoftkeyword`.

**Environment/tooling issue (not a code defect) — repeated silent write corruption on the delivery mount.**
During final delivery, three separate files (`ts_corpus/package.json`, `cli.py` twice, `tests/test_ts_cli_corpus.py`) were silently truncated or null-padded after `Edit`-tool operations, in a way that was invisible until caught by re-running the test suite / `npm install` against the actually-delivered files rather than trusting the tool's success response. This matches a previously-logged environment gotcha (mount write quirks) but is more severe than previously observed — it is not just "stale read," it is content loss that persisted across retries and even fresh full-file `Write` calls to the SAME path, and only resolved by writing to a new filename and renaming over the original. **Every file in this delivery was re-verified against the actual mounted files** (full test suite, `npm install`, `pip-audit`, corpus eval, and a manual null-byte/truncation scan) after this was discovered — the numbers in Section 2 are from that final, post-fix verification pass, not from the sandbox dev copy. Flagged in Section 6 as this session's suggested process improvement.

## 4. Known Limitations (stated honestly)

- **Module augmentation (`declare module "..." { interface X {...} }`) is not merged — the whole package degrades.** This is the single biggest real-world gap: `@types/lodash` (and much of DefinitelyTyped) attaches most of its public API to an already-declared interface via augmentation blocks scattered across a dozen files, which is TypeScript declaration-merging semantics this KB does not attempt to re-implement. Verified end-to-end against the real, installed `@types/lodash` package: it correctly degrades (never flags `_.debounce(...)` or anything else from it), but this also means a genuinely hallucinated lodash call is silently invisible to this engine. This is the exact risk Yehor accepted by choosing tree-sitter + custom KB over the TS Compiler API, now concretely measured rather than theoretical.
- **Deep subpath imports (`lodash/debounce`) are only checked for "package installed," not symbol existence.** This KB only parses a package's declared ENTRY `.d.ts`; a subpath's own declaration file is not located or parsed. A hallucinated deep-import target is not caught.
- **Cross-package re-exports are not followed.** Only relative-path re-exports within the same package are resolved transitively (fixed this session, see DEFECT KB-002); a re-export whose source is another npm package degrades that one name only, not the whole package.
- **`implements` clauses do not contribute members.** Only `extends` chains (interface-extends-interface, interface-extends-class, class-extends-class) are followed for member flattening.
- **A default-import binding is never flagged even with no resolvable default export**, because `default_export_symbols is None` is ambiguous between "genuinely no default export" and "type exists but couldn't be flattened" (e.g., a function type or primitive, not an interface/class). Resolves to silence per the zero-false-positive mandate — a real recall cost, stated plainly.
- **No `dependency-version-mismatch` reason code yet on the TS side** — there is no npm lockfile/version-pin cross-check this session, matching the Python resolver's own original Stage-1 scope for that reason code.
- **Generic type arguments on `extends` targets are ignored** (`extends Base<T>` resolves to `Base` by name only) — sufficient for existence checks, not deeper generic reasoning.
- **The corpus's real-package coverage is necessarily narrower than the Python corpus's.** Only `axios` and (for the degrade case) `lodash`/`@types/lodash` are real, installed packages; the re-export/extends-chain/type-only-import adversarial cases run against a hand-built synthetic fixture (`fp-ts-clean-demo`) because the sandbox's available real packages didn't happen to exercise multi-file re-export chains as cleanly as a purpose-built fixture can. This mirrors Session 1.3's own corpus methodology (real packages for the flagship cases, synthetic for controlled edge cases), stated explicitly rather than implied.

## 6. Methodology Note — One Suggested Improvement

This session's real, load-bearing lesson is procedural, not architectural: **"the tool reported success" is not sufficient evidence that a delivered file is correct on the target filesystem.** Three separate files were silently truncated or null-padded by `Edit`/`Write` operations against this specific mounted path, in ways invisible until the full test suite (or `npm install`, or a manual `tail`/`stat`) was re-run against the actual delivered files rather than the sandbox's own working copy. The fix that worked reliably was writing to a fresh filename and renaming over the target — never re-editing the same already-corrupted path in place.

Suggested process improvement for every future session (not just this one): **treat "delivered" and "verified" as genuinely separate steps, in that order, every time** — after copying/writing final files to the delivery location, always re-run the full test suite (and, for anything with an external package manager, the actual install command) directly against the delivered path before writing the Keystone Report, even when the dev-sandbox copy already passed. This session's Section 2 numbers are trustworthy specifically because that re-verification was done; a version of this report written before that final pass would have contained numbers matching a subtly different (and in cli.py's case, silently mis-instrumented) file than what Yehor would actually receive.

A secondary, smaller suggestion: Session 1.3's `corpus/requirements.txt` has the same latent reproducibility gap this session found and fixed on the TS side (a deliberately-uninstalled package name lives directly in the manifest file) — if anyone ever runs `pip install -r corpus/requirements.txt` to bootstrap that corpus, it will fail the same way `npm install` did here before the fix. Not fixed in this session (out of scope, no sign-off to touch Session 1.3's files), but worth a one-line fix whenever that corpus is next touched.
